# Rule of Zero

Ако имаме много класове като:

```cpp
class Person {
    char* name;
};

class Car {
    char* model;
};

class Book {
    char* title;
};
```

Всеки един от тях:
- ръчно управлява памет
- има една и съща логика
- изисква писане на destructor, copy/move
- има риск от грешки (memory leak, double free, shallow copy)

Това може да е лош дизайн.

## RAII (Resource Acquisition Is Initialization)

Обектът винаги е във валидно състояние и сам се грижи за ресурса си.

- ресурсът се придобива в конструктор
- ресурсът се освобождава в деструктор
- ресурсът има ясен собственик (обект)

```
class MyString {
    char* data;
public:
    MyString(const char* str);
    ~MyString(); // освобождава паметта
};
```

Вместо всеки клас да управлява памет:

```c++
class Person {
    char* name;
};
```

използваме RAII клас:
```c++
class Person {
    MyString name;
};
```

или стандартен:

```c++
class Person {
    std::string name;
};
```

Така ```Person``` не управлява памет, а ```MyString``` / ```std::string``` го прави като имаме една точка на отговорност.

## Rule of Zero

- не пишем деструктор
- не пишем copy/move функции
- оставяме компилатора да ги генерира

```c++
class Person {
    МyString name;
};
```

- няма ръчно управление на "външни" ресурс
- няма нужда от специални функции


Можем да имаме един RAII клас, който управлява ресурса отколкото много класове с еднакви указатели и логика.

Но често в реалния свят има много случаи и варианти за имплементация на структури от данни(като string, vector и тн.) и управлението на ресурси може да е сложно и трудно за откриване в различните сценарии. Затова стандартните контейнери (std::string, std::vector и т.н.) са достатъчно добре оптимизирани и често е по-добре да ги използваме наготово. 

## Кога има смисъл да пишем собствен клас за СД, които вече имат предоставена имплементация?

Може да дефинираме собствен RAII клас, ако:
- имаме конкретен ресурс
- използваме го с една ясна цел
- искаме контрол върху управлението му

Не е добра идея, ако структурата ще се използва:
- в много различни контексти
- за различни цели
- с различни изисквания

## Задачи
**Задача 1:**
Реализирайте клас **MulString**, предвиден за работа със символни низове, който да поддържа следните оператори:
- оператор *= , който по подадено цяло число k мултиплицира съдържанието на даден стринг k пъти
- оператор * , който по подадени стринг и число връща стринг, чието съдържание е мултиплицираното k пъти съдържание на поддадения стринг
- оператор за записване в поток <<
- оператор != , сравнява два стринга по тежест, връща true <=> двата стринга са с различна тежест
- оператор %= , който като резултат дава обеднинението без повторения на даден стринг с друг
- оператор % , който като резултат дават обеднинението без повторения на два стринга

**Упътване:**
Обединение без повторения на два стринга дефинираме като: Всички символи , които се съдържат в първия и втория стринг, записани еднократно. Наредбата в резултата не е от значение.
Пример : ABCDEFABJ % BCDGHJ -> ABCDEFJGH

Тежест на стринг : сумата от ASCII кодовете на символите на стринга.

Демонстрирайте използването на операторите в кратка main функция
